Azure - NetSuite REST Interface

About

This documentation provides a comprehensive overview of an Azure Function application built to interface seamlessly with NetSuite's REST API. Designed for performance and scalability, the application minimizes computational overhead while enabling secure, real-time access to NetSuite data. Leveraging serverless architecture and optimized request handling, it supports efficient execution of SuiteQL queries and CRUD operations. Ideal for automation workflows, system integrations, and data-driven services, this solution eliminates the need for complex middleware and delivers a lightweight, maintainable interface for enterprise environments.

Key Features

  • Endpoint Integration: Dedicated endpoints for executing SuiteQL queries and performing CRUD operations.
  • Authentication: Secure access using an authentication header (x-functions-key).
  • Query Execution: Dynamic data retrieval from NetSuite.
  • Continuous Improvement: CI/CD Pipeline for continuous updates and enhancements with the ability to push updates to client Azure environments through Github Actions.

Use Cases

  • Integration: Integrate NetSuite data with other Azure services.
  • Data Retrieval: Extract detailed information from NetSuite.
  • Record Management: Create, update, and delete NetSuite records available via REST API.
  • Data Transformation: Convert records from one type to another (e.g., Sales Order to Item Fulfillment, Sales Order to Invoice).
  • Automation: Automate routine NetSuite operations.

Benefits

  • Solution Cost: Extremely low operational overhead.
  • Scalability: Built on Azure Functions, allowing for automatic scaling based on demand.
  • Security: Built-in authentication and secure communication protocols.
  • Flexibility: Execute custom SuiteQL queries.
  • Efficiency: Streamlined data access and manipulation through RESTful endpoints.
  • In House Development: Developed in-house, allowing for tailored solutions to meet specific business needs.

Getting Started

All endpoints require an x-functions-key header for authentication. This key is generated by Azure Functions and can be retrieved from the Azure Portal under your Function App → Functions → select the function → Function Keys. Contact your Azure administrator if you do not have portal access.

Every endpoint URL contains the placeholder {companyName}. Replace this with your Azure Function App name (the subdomain of your .azurewebsites.net deployment). For example, if your app is deployed at nsrestinterface-acmecorp.azurewebsites.net, use acmecorp as the value.

Function: netsuiteSuiteqlQuery

Endpoint: https://nsrestinterface-{companyName}.azurewebsites.net/api/netsuiteSuiteqlQuery

Authentication: Required (x-functions-key)

Parameters

ParameterTypeRequiredDescription
querystringYesSuiteQL query string to execute.
nextPageLinkstringNoOptional. If paginating, pass the nextPageLink value from a previous response to retrieve the next page of results.

Notes

  • Query must be valid SuiteQL syntax.
  • Authentication header x-functions-key is required.
  • If the result set exceeds 1000 records, the response will include a links array with objects containing rel (such as next, last, self) and href properties. While hasMore is true, pass the href value from the rel: next object as nextPageLink in your next request payload to fetch subsequent pages.

Ex 1: Basic Query

Method: POST

URL: https://nsrestinterface-{companyName}.azurewebsites.net/api/netsuiteSuiteqlQuery

Headers: x-functions-key

Payload:

{
    "query": "select id, trandate, entity from transaction where type = 'SalesOrd' and trandate >= '1/1/2025'"
}

Success Response:

{
    "items": [
        { "id": "1001", "trandate": "1/15/2025", "entity": "100" },
        { "id": "1002", "trandate": "2/11/2025", "entity": "101" }
    ],
    "count": 2,
    "hasMore": false
}

Ex 2: Invalid Query (Error)

Payload:

{
    "query": "select * from non_existing_table"
}

Error Response:

{
    "type": "https://www.rfc-editor.org/rfc/rfc9110.html#section-15.5.1",
    "title": "Bad Request",
    "status": 400,
    "o:errorDetails": [
        {
            "detail": "Invalid search query. Table does not exist.",
            "o:errorQueryParam": "q",
            "o:errorCode": "INVALID_PARAMETER"
        }
    ]
}

Ex 3: Advanced Query (Join)

Payload:

{
    "query": "select t.id, t.trandate, c.entityid from transaction t join customer c on t.entity = c.id where t.type = 'SalesOrd' and c.email like '%@example.com'"
}

Success Response: (truncated)

{
    "items": [
        { "id": "1002", "trandate": "2/11/2025", "entityid": "New Customer" },
        ...
    ],
    "count": 2,
    "hasMore": false
}

Ex 4: Empty Result Set

Payload:

{
    "query": "select * from transaction where id = -1"
}

Success Response:

{
    "items": [],
    "count": 0,
    "hasMore": false
}

Ex 5: Paginated Multi-Page Query

Payload (Page 1):

{
    "query": "select id from transaction"
}

Success Response (Page 1):

{
    "links": [
        { "rel": "next", "href": "https://.../suiteql?limit=1000&offset=1000" }

Error Handling

Authentication Errors

A missing or invalid authentication 'x-functions-key' passed in the API call will result in a 401 Unauthorized error code response.

Improperly Formatted Requests

Queries or Post Requests that do not contain valid JSON will result in a 500 error code response. Ensure all objects are properly wrapped and formatted before proceeding.

SuiteQL & REST Requests

All NetSuite error messages are returned from the server upon a failed API call, along with a 400 error code response. The format of the messages is inherited from the NetSuite server and typically resembles the below message.
